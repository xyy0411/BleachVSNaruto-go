# AI 代码审查规则

本仓库使用 Go 语言编写,是使用ebiten开发的2D横板格斗游戏

审查拉取请求时，请遵循以下规则

---

# 文档

所有导出的标识符必须有文档注释

规则：

1. 每个导出的函数、类型、方法或变量必须有注释
2. 注释必须以标识符名称开头（Go 文档风格）
3. 新注释的首选语言是中文
4. 如果贡献者已经用英文写了注释，不要要求翻译
5. 注释应清楚解释函数的功能
6. 如果有帮助，可以简要描述参数和返回值
7. 对如果已经存在的注释，请勿修改(如果真的是有错误，请修改)
8. 每个注释必须要言简意赅的回答不能有任何的冗余问题,也不能答非所问.如果这个函数或者方法的职责很简单或者是函数名已经能概括职责则注释可概括为...
9. 每个函数和方法最起码都要有一个简单的注释

示例：

```
// RegisterCharacter 注册一个新的角色到角色注册表中
func RegisterCharacter(name string) error
```
避免这样的注释：

```
// 注册角色
func RegisterCharacter(...)
```

因为 Go 文档注释应以标识符名称开头

要求第八条的描述：

```
// Update 更新系统 ×
// Update 字段s.Current在此方法里已经代表为上一帧发生的事件了
// 所以我们可以通过前一帧发生的事件来处理本帧应该处理的事件 √
func (s *controller.System) Update()
```
---

# 代码风格

遵循惯用的 Go 语言风格

一般原则：

- 偏好简单且可读的代码
- 避免不必要的抽象
- 避免过于复杂的逻辑
- 保持函数职责单一
- 偏好清晰而非巧妙

偏好这种风格：
`
if err != nil {
return err
}
`

避免深层嵌套的逻辑

---

# 错误处理

错误应该始终被显式处理

规则：

- 不要忽略返回的错误
- 不要使用空错误检查
- 倾向于返回错误而不是隐藏它们
- 错误信息应有意义

---

# 命名

名称应清楚描述其用途

避免模糊的名称，如：

DoThing
HandleStuff
ProcessData
Manager2

偏好描述性名称，如：

LoadCharacterData
RegisterPlayer
HandlePlayerInput

---

# 代码质量

审查代码时，优先考虑：

1. 缺少文档注释
2. 被忽略的错误
3. 潜在的 bug
4. 令人困惑的逻辑
5. 糟糕的命名
6. 不必要的复杂性

如果可能，建议更清晰的实现

---

# 性能

避免不必要的内存分配

偏好简单的数据结构

不要过早优化，但要指出明显的低效问题

---

# 并发（如果存在）

当使用 goroutine 或共享数据时，检查：

- 竞态条件
- 不安全的共享状态
- 缺少同步机制

当需要时，建议更安全的模式

---

# 建议

当发现问题时：

- 清楚地解释问题
- 建议更好的实现方式
- 如果有帮助，提供示例代码
- 注释中不要添加类似反人类的`。`

保持建议简洁且可操作

如果代码已经遵循仓库规则且没有发现问题，
只需回应 👍
